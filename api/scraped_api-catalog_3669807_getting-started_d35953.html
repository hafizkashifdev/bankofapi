Getting Started - Age Verificaton

Key Sandbox endpoints

  • Well-known: https://api.sandbox.natwest.com/.well-known/openid-configuration
  • Token: https://ob.sandbox.natwest.com/token
  • Authorization: https://api.sandbox.natwest.com/authorize
  • Resources: https://ob.sandbox.natwest.com/zerocode/bankofapis.com/customer-age/v3/attributes/age

Assumptions

  • You have already registered an App in your Team
  • Or you use the demo one
  • Your app is using the 'Transport and signing certificates - Private key JWT' authentication method

Goal

This section aims at giving you a comprehensive step by step tutorial to guide you through the process of configuring the usage and executing the Age Verification API by bankofapis.com

Postman

If you are a Postman user, you can easily explore and run the steps of this tutorial using our predefined collection and environment files:
Note that in Postman you have to go in Preferences or Settings > General menu
  • in REQUEST column, make sure that Ssl certificate verification is set to OFF (or make sure that the OpenBanking Pre-Production Issuing CA is in your trusted store)
  • in HEADERS column, make sure that Automatically follow redirects is set to ON

Download and import in Postman
  • Download your bankofapis.com/customer-age Postman collection
  • Download your Postman environment files per Application below:
    • demo-app-3c216362-f073-42f2-8985-af4031365d94
Once you have imported the collection and environment, make sure the client_id and client_secret values in the environment file match your Client ID and Client Secret, which can be found inside your App dashboard.

Steps

1. Setup account access consent

Retrieve access token

To create an account-access-consent, you'll need to pass an access-token identifying your App. Simply replace CLIENT_ID and CLIENT_SECRET in the curl below to retrieve a valid one:
Depending on the language you use, generating and signing this JWT would be different. A non-binding example of how to do this using the RSA-Sign JavaScript Library could look like: 
let nowInSeconds = Math.round(Date.now() / 1000);

var requestData = {
  "jti": <unique id>,
  "aud": <token endpoint>,
  "iss": <clientId>,
  "sub": <clientId>,
  "exp": (nowInSeconds + (60 * 5))
};

var header = {
  "typ": "JWT",
  "kid": <kid that has been declared in the Sandbox for this Signing certificate, "unknown" if none was specified>
};

var signingPrivateKeyAsPem = <private key associated with the Signing certificate that has been declared in the Sandbox>;

var sHeader = JSON.stringify(header);
var sPayload = JSON.stringify(requestData);
var generatedJwt = KJUR.jws.JWS.sign("PS256", sHeader, sPayload, signingPrivateKeyAsPem);
Request: 
curl -k -X POST \
  https://ob.sandbox.natwest.com/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d 'grant_type=client_credentials&client_id=CLIENT_ID&client_secret=CLIENT_SECRET&scope=accounts customer-age verify-age'
Response 
HTTP/1.1 200
status: 200
content-type: application/json; charset=utf-8

{
    "token_type": "Bearer",
    "access_token": "Ci5A6huK2OD0d6...kVwLE9FuHwHGCf6xo5",
    "expires_in": 7200,
    "scope": "accounts customer-age verify-age"
}
Your access-token is the value of the access_token attribute in the Response. In the example above it would be Ci5A6huK2OD0d6...kVwLE9FuHwHGCf6xo5 .

Submit account access consent

Now you can create an account-access-consent by replacing the ACCESS_TOKEN in the curl below.
Request: 
curl -k -X POST \
  https://ob.sandbox.natwest.com/open-banking/v3.1/aisp/account-access-consents \
  -H 'Authorization: Bearer ACCESS_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{
  "Data": {
    "Permissions": [
      "ReadAccountsDetail"
    ]
  },
  "Risk": {}
}'
Response: 
HTTP/1.1 201
status: 201
content-type: application/json;charset=utf-8

{
  "Data": {
    "Permissions": [
      "ReadAccountsDetail",
    ],
    "ConsentId": "69ae997b-fe70-45eb-a00d-c49a1d310b51",
    "CreationDateTime": "2019-03-06T16:01:17.876Z",
    "Status": "AwaitingAuthorisation",
    "StatusUpdateDateTime": "2019-03-06T16:01:17.876Z"
  },
  "Risk": {},
  "Links": {
    "Self": "https://ob.sandbox.natwest.com/open-banking/v3.1/aisp/account-access-consents/69ae997b-fe70-45eb-a00d-c49a1d310b51"
  },
  "Meta": {}
}
When successful, in the response you'll find a consent-id identifying your request in the Data.ConsentId attribute that you will have to reference in the next step. In the example above it would be 69ae997b-fe70-45eb-a00d-c49a1d310b51 .

3. Authorize consent

Once your account-access-consent is created, you need to make the user approve it by redirecting their browser to a customized authorization url.

Construct authorization request url

There are two options:
"Reduced security" mode
You need to activate this mode by ticking the "Allow reduced security" box when you are requesting access to CMA9 account api for an App in the developer portal.
In this mode, the consent-id identifying the account-access-consent submitted previously is passed as-is in the authorization url. It can be achieved by replacing the CLIENT_ID , REDIRECT_URI and CONSENT_ID in the curl below:
The example is displayed on multiple lines for readability but it should be used as a one liner: 
  https://api.sandbox.natwest.com/authorize
    ?client_id=CLIENT_ID
    &response_type=code id_token
    &scope=openid accounts customer-age verify-age
    &redirect_uri=REDIRECT_URI
    &request=CONSENT_ID
The postman collection constructs the authorization url like so under the visualize section of the response, where it can be copied to clipboard and pasted into your browser.
Standard mode
Please refer to the extensive documentation of Security Profile for the details on how to construct the request parameter of the authorize call: Hybrid Flow Request documentation

User approval

The user is redirected to a browser to approve consent. For this step, you will temporarily be impersonating the user.
  • Paste the url in a new browser window and press Enter. The login page of the bank is displayed
  • Follow the authentication flow by filling in one of your test user credentials or use the demo one (123456789012 as Customer number and type the digit above each inputs for pin/password)
  • The credentials are verified and if correct you should be presented with a page displaying the details of the consent to approve
  • Verify the content, select the accounts you want to give access to and click 'Confirm access'
  • The url in the address bar of your browser should now contain an url like this: 
  REDIRECT_URI#code=nO93oFbrPGvyeMZSIQFq3xLYWODp1Oi5&id_token=...
This url contains a fragment parameter code containing the authorization code you must exchange for an access-token in the next step. In the example above it would be nO93oFbrPGvyeMZSIQFq3xLYWODp1Oi5.

Exchange code for access token

Now that you received an authorization code, you need to exchange it for an access token. It can be simply achieved by replacing the CLIENT_ID , CLIENT_SECRET , REDIRECT_URI and AUTHORIZATION_CODE in the curl below:
Request: 
curl -k -X POST \
    https://ob.sandbox.natwest.com/token \
    -H 'Content-Type: application/x-www-form-urlencoded' \
    -d 'client_id=CLIENT_ID&client_secret=CLIENT_SECRET&redirect_uri=REDIRECT_URI&grant_type=authorization_code&code=AUTHORIZATION_CODE'
Response: 
HTTP/1.1 200
status: 200
content-type: application/json; charset=utf-8

{
"refresh_token": "ZzBmtBMhC70Y...95LugYlvyUEyXQLXk34Q",
"token_type": "Bearer",
"access_token": "tHIYRoTsIL2tb...hwvK29ZKsoJGXFKINzP",
"id_token": "eyJhbGciOiJQ...UzI1NiIsInR5cCI6IkpXVCJ",
"expires_in": 7200,
"scope": "openid accounts customer-age verify-age"
}
Your access-token is the value of the access_token attribute in the Response. In the example above it would be tHIYRoTsIL2tb...hwvK29ZKsoJGXFKINzP .
To exhange the authorization code for an access-token in Postman, you can copy the authorization code into the field code , which is found in the body of the call.

4. Request data

Finally you can perform api calls to retrieve information about the bank accounts for which the user approved the consent. For example you can perform the following call to retrieve age for authorised customers (since we initially asked for this level of permission in our account-access-consent) by replacing the ACCESS_TOKEN in the curl below:
Request: 
curl -k -X GET \
  https://ob.sandbox.natwest.com/zerocode/bankofapis.com/customer-age/v3/attributes/age \
  -H 'Authorization: Bearer ACCESS_TOKEN'
